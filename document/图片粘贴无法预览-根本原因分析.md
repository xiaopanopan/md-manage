# 图片粘贴后无法预览 — 根本原因分析

> 日期：2026-04-18
> 版本：v0.2.1 基线
> 症状：编辑器中粘贴图片后，切换到阅读模式图片无法显示（显示为破图或占位）

---

## 一、完整流程链路

### 1. 粘贴触发（`EditorExtensions.ts` → `imageHandlers`）

```typescript
paste(event, view) {
  const items = Array.from(event.clipboardData?.items ?? []);
  const img = items.find(i => i.type.startsWith('image/'));
  event.preventDefault();
  (async () => {
    const blob = img.getAsFile();
    const buffer = Array.from(new Uint8Array(await blob.arrayBuffer()));
    const ext = img.type.split('/')[1] || 'png';  // ⚠️ 问题点 #1
    const relPath = await window.electronAPI?.image.save(buffer, ext);
    if (relPath) insertText(view, `![](${relPath})`);  // ⚠️ 问题点 #2
  })();
}
```

### 2. 主进程保存（`fileSystem.ts` → `image:save`）

```typescript
ipcMain.handle('image:save', async (_event, buffer, ext) => {
  const workspace = getConfig('workspace');
  const imagesDir = path.join(workspace, '.digitalzen', 'images');
  await fs.mkdir(imagesDir, { recursive: true });

  const filename = `${Date.now()}.${ext.replace(/^\./, '')}`;
  const absPath = path.join(imagesDir, filename);
  await fs.writeFile(absPath, Buffer.from(buffer));

  return path.join('.digitalzen', 'images', filename);  // ⚠️ 问题点 #3
});
```

### 3. 编辑器写入

插入的 Markdown 文本：`![](.digitalzen/images/1718000000000.png)`

### 4. 阅读模式渲染（`markdown.ts` → `rehypeFixImagePaths`）

```typescript
function rehypeFixImagePaths(workspace) {
  return (tree) => {
    visit(tree, 'element', (node) => {
      if (node.tagName === 'img' && node.properties?.src) {
        const src = node.properties.src;
        if (!src.startsWith('http') && !src.startsWith('file://') && !src.startsWith('data:')) {
          node.properties.src = `file://${workspace}/${src}`;  // ⚠️ 问题点 #4
        }
      }
    });
  };
}
```

### 5. HTML 输出 + Sanitize

```html
<img src="file:///Users/mlamp/Documents/工作区/.digitalzen/images/1718000000000.png">
```

Sanitize schema 允许的 src 正则：`/^(https?:|file:)/` → 匹配 `file:` 开头 ✅

### 6. Electron 加载 `file://`

`electron/main.ts`：
```typescript
webPreferences: {
  ...
  webSecurity: false,   // ✅ 已禁用跨域检查
}
```

`index.html` CSP：
```
img-src 'self' data: file:;   // ✅ 允许 file: 协议
```

## 二、根本原因排查（按优先级）

### 🔴 P0：路径未做 URL 编码（最可能原因）

**位置**：`src/lib/markdown.ts:31`

```typescript
node.properties.src = `file://${workspace}/${src}`;
```

**问题**：直接字符串拼接，完全没有 URL 编码。一旦工作区路径或文件名含以下字符之一，生成的 `file://` URL 即无效：

| 字符 | 示例路径 | 生成的 URL | 结果 |
|------|----------|-----------|------|
| 空格 | `/Users/mlamp/Documents/My Docs` | `file:///Users/mlamp/Documents/My Docs/...` | ❌ 浏览器视为多个 URL |
| 中文 | `/Users/mlamp/Documents/工作区` | `file:///Users/mlamp/Documents/工作区/...` | ⚠️ 部分环境失败 |
| `#` | `/Users/mlamp/Docs#1/foo` | `file:///Users/mlamp/Docs#1/foo/...` | ❌ `#1` 被当作 hash |
| `?` | `/Users/mlamp/Docs?/foo` | `file:///Users/mlamp/Docs?/...` | ❌ `?` 被当作 query |

**修复**：
```typescript
const encoded = src
  .split('/')
  .map(encodeURIComponent)
  .join('/');
const encodedWorkspace = workspace
  .split('/')
  .map(encodeURIComponent)
  .join('/');
node.properties.src = `file://${encodedWorkspace}/${encoded}`;
```

---

### 🟡 P1：Windows 路径分隔符（跨平台问题）

**位置**：`electron/ipc/fileSystem.ts:183`

```typescript
return path.join('.digitalzen', 'images', filename);
```

**问题**：`path.join` 在 Windows 返回 `.digitalzen\images\xxx.png`（反斜杠）。但 Markdown 语法 `![](path)` 中反斜杠是转义字符，会破坏路径解析。

**macOS 不受影响**（`/` 分隔符），Windows 用户会遇到。

**修复**：
```typescript
return ['.digitalzen', 'images', filename].join('/');
// 或
return path.posix.join('.digitalzen', 'images', filename);
```

---

### 🟡 P1：扩展名推断错误

**位置**：`EditorExtensions.ts:167`

```typescript
const ext = img.type.split('/')[1] || 'png';
```

**问题**：MIME 类型 → 扩展名的映射不准确：

| MIME 类型 | `split('/')[1]` | 正确扩展名 |
|-----------|-----------------|-----------|
| `image/png` | `png` | ✅ `png` |
| `image/jpeg` | `jpeg` | ⚠️ 通常是 `jpg` |
| `image/svg+xml` | `svg+xml` | ❌ 应为 `svg`（`+` 也不是合法文件名字符） |
| `image/webp` | `webp` | ✅ `webp` |
| `image/gif` | `gif` | ✅ `gif` |

文件名含 `+` 虽然在文件系统是合法的，但加上 URL 未编码时会造成解析歧义。

**修复**：
```typescript
const MIME_EXT_MAP: Record<string, string> = {
  'image/png': 'png',
  'image/jpeg': 'jpg',
  'image/gif': 'gif',
  'image/webp': 'webp',
  'image/svg+xml': 'svg',
  'image/bmp': 'bmp',
};
const ext = MIME_EXT_MAP[img.type] || 'png';
```

---

### 🟢 P2：CSP 与 `webSecurity` 交互（低概率）

**位置**：`index.html:8`

```
img-src 'self' data: file:;
```

**分析**：
- CSP `file:` 匹配任何 `file:` 开头的 URL ✅
- Electron 的 `webSecurity: false` 允许跨域加载 ✅
- Dev 环境页面从 `http://localhost:5173` 加载，加载 `file://` 资源属于跨 scheme

Electron 28 版本 webSecurity: false 可正常工作，此处应不是问题。**排除。**

---

### 🟢 P2：文件未实际写入（低概率）

**排查路径**：
```bash
ls -la "<workspace>/.digitalzen/images/"
```

若目录存在文件但大小为 0 → Blob 数据丢失
若目录不存在 → IPC 未执行成功

**可能失败场景**：
- 工作区尚未选择（handler 会抛 `No workspace configured` 错误）
- 剪贴板 MIME 类型声明为 image 但 `getAsFile()` 返回 null
- `webSecurity: false` + `sandbox: false` 环境下，`arrayBuffer()` 偶发失败

## 三、验证清单

### 快速定位实际问题

1. **检查生成的 Markdown 引用**
   - 切换到编辑模式，粘贴后查看插入的文本
   - 确认是 `![](.digitalzen/images/xxxxx.png)` 格式
   - ✅ 正常 → 继续第 2 步
   - ❌ 无插入 → IPC 调用失败，查看主进程 console

2. **检查文件是否写入磁盘**
   ```bash
   ls -la "<workspace>/.digitalzen/images/"
   ```
   - ✅ 有文件 → 继续第 3 步
   - ❌ 无文件 → `image:save` handler 异常

3. **检查阅读模式渲染的 HTML**
   - 打开 DevTools (Cmd+Option+I)
   - 切换到阅读模式
   - Elements 面板查看 `<img>` 标签的 src
   - 复制 src，在地址栏新标签页打开
   - ✅ 能预览 → Electron 加载正常，可能是 CSS 隐藏或其他问题
   - ❌ 404 / 无法访问 → URL 构造有问题（P0 路径编码）

4. **检查 Network 面板**
   - DevTools → Network → 筛选 Img
   - 切换到阅读模式，看图片请求
   - Status 如果是 `(failed) ERR_FILE_NOT_FOUND` → URL 路径错
   - Status 如果是 `(failed) net::ERR_BLOCKED_BY_CSP` → CSP 问题
   - Status 如果没有请求 → HTML 根本没输出 `<img>` 标签（sanitize 过滤掉？）

## 四、推荐修复方案

按修复优先级排序：

### Fix 1（必改）：URL 编码 + 保留路径分隔符

```typescript
// src/lib/markdown.ts
function encodeFilePath(p: string): string {
  return p
    .split('/')
    .map((seg) => encodeURIComponent(seg))
    .join('/');
}

function rehypeFixImagePaths(workspace: string) {
  return (tree: AnyNode) => {
    visit(tree, 'element', (node: AnyNode) => {
      if (node.tagName === 'img' && node.properties?.src) {
        const src = node.properties.src as string;
        if (!src.startsWith('http') && !src.startsWith('file://') && !src.startsWith('data:')) {
          const encodedWs = encodeFilePath(workspace);
          const encodedSrc = encodeFilePath(src.replace(/\\/g, '/'));  // Windows 反斜杠转正斜杠
          node.properties.src = `file://${encodedWs}/${encodedSrc}`;
        }
      }
    });
  };
}
```

### Fix 2（建议）：统一使用正斜杠的相对路径

```typescript
// electron/ipc/fileSystem.ts
ipcMain.handle('image:save', async (_event, buffer, ext) => {
  // ...
  return `.digitalzen/images/${filename}`;  // 强制正斜杠
});
```

### Fix 3（建议）：MIME 到扩展名的正确映射

```typescript
// src/components/editor/EditorExtensions.ts
const MIME_TO_EXT: Record<string, string> = {
  'image/png': 'png',
  'image/jpeg': 'jpg',
  'image/gif': 'gif',
  'image/webp': 'webp',
  'image/svg+xml': 'svg',
  'image/bmp': 'bmp',
};

const ext = MIME_TO_EXT[img.type] ?? 'png';
```

## 五、结论

**最可能的根本原因**：**P0 路径未 URL 编码**

只要用户工作区路径包含空格、中文或特殊字符（非常常见），生成的 `file://` URL 就是无效的，浏览器无法加载。Electron 的 webSecurity、CSP 都没问题，问题出在 URL 生成环节。

**立即验证方法**：打开 DevTools Elements 面板，复制某个失败的 `<img src="file://...">` 的完整 URL，直接粘贴到新标签页看能否访问。如能看到 URL 中有未编码的空格/中文，即可确认。
